# Полный технический аудит и архитектурное ревью

> Комплексный аудит проекта Brevity как долгоживущей системы (поддержка и развитие
> на годы вперёд). Дата: 2026-06-05.
> Стек: PHP 8.4 / Laravel 13.14 / PostgreSQL / Filament v5.6 / Sanctum / Horizon /
> Octane (FrankenPHP) / Hashids / `vaslv/laravel-settings`.

**Ревьюеры:** Senior Laravel / Filament / PostgreSQL Architect + Principal Engineer,
**Codex** — независимый внешний валидатор.

**Метод.** 9 параллельных агентов-картографов прочитали весь код (~700k токенов,
452 вызова инструментов) → структурная карта подсистем со 143 «подозрениями» →
13 состязательных проверок → кросс-валидация Codex → 5 оценок по измерениям.
**Каждая ключевая находка воспроизведена на живом приложении/БД**, а не выведена из
чтения кода.

> **Контекст severity.** `docs/ADMIN.md` фиксирует инвариант: *каждый
> аутентифицированный `User` — полностью доверенный администратор*. Находки,
> требующие уже-доверенного админа, оцениваются как **defense-in-depth**, а не
> критичные. Находки, достижимые **неаутентифицированным** посетителем или
> **внешним держателем API-токена**, оцениваются по реальной поверхности атаки.

задать `HEALTHCHECK_URL` (напр. `https://<APP_HOST>/up`); включить retention

вырастут.

H4 — соль вынесена); SMTP-креды и `HASHIDS_SALT` в серверном `.env`; включить
`DB_SSLMODE=require`, когда БД будет с TLS.

## Executive summary

Brevity — **хорошо спроектированный, продуманный код**, явно прошедший дисциплинированный
цикл ревью (`docs/CODE_REVIEW.md`). Сервисный слой когерентный и `readonly`,
идемпотентность встроена, а не прикручена; SSRF-гард аккуратно обоснован; Octane-гигиена
чистая (проверено — **утечки** состояния между запросами **нет**); культура регресс-тестов
(один файл на исторический баг, red-test-then-fix) образцовая. **Продолжать разработку
безопасно.**

Чего прошлое ревью **не** заметило — слой **латентных проблем масштабирования и
устойчивости**, всплывающих только под нагрузкой или на враждебном вводе, плюс одно
действительно ошибочное «закрытое» утверждение.

| Измерение | Оценка | Кратко |
|---|---|---|
| Архитектура / качество кода | **B** | Сильные сервисные швы; избыточная индирекция «трейт на связь»; бизнес-критичная запись спрятана в boot-хуке модели. |
| Производительность / масштаб | **C** | Здоровое асинхронное ядро, но некэшируемый горячий путь, 2 недостающих индекса, не-sargable запросы дашборда, **нулевой retention**. |
| Безопасность | **B** | Сильное моделирование угроз; тянут вниз **две реальные неаутентифицированные** бреши + DiD-зазоры, которые держит только инвариант доверенного админа. |
| Filament / админ-UX | **B** | Идеально единообразная структура; «сырые» ошибки Postgres закрыты лишь в 2 из ~7 мест. |
| Тестирование | **C+** | Отличная дисциплина, зелёный базлайн — но **ядро резолвинга практически не покрыто**, и нет фабрик. |

- **Сопровождаемость:** хорошая. Чёткие конвенции, сильная документация, тонкие контроллеры.
- **Масштабируемость:** редирект-путь асинхронно-корректен, но латентность **монотонно
  деградирует** по мере роста `clicks`/`callbacks` до миллионов.
- **Безопасность:** хорошая для внутреннего инструмента. Две реальные внешние бреши + набор
  DiD-зазоров.
- **Техдолг:** **низкий-умеренный.** Нет god-классов, злоупотреблений фреймворком, гнили.
  Долг сосредоточен в устойчивости схемы БД, гигиене деплоя и покрытии ядра тестами.

**Счёт:** Critical **2** · High **8** · Medium **15** · Low **~18** · Positives — много.

**Вердикт Codex:** согласен со всеми выводами, не опроверг **ни одного**. Две поправки
вплетены ниже (более широкий остаточный риск `172.16/12`; комментарий «race-safe on its
own» — то же заблуждение, что оставило `callbacks.click_id` без защиты).

## 🔴 Critical

### C1 — Неаутентифицированная потеря кликов через переполнение btree-индекса Postgres
- **Категория:** Security / Reliability · **Приоритет:** немедленно
- **Описание:** `referrers.value`/`urls.value` — `varchar(2048) UNIQUE`, `user_agents.value` —
  `text UNIQUE`, все обычные btree. Приложение режет значения по **числу символов**
  (`ClickRecorder`: referrer 2048, UA 1024 через `Str::limit`). Лимит btree-ключа Postgres —
  **2704 байта**. Несжимаемый многобайтовый заголовок `Referer`/`User-Agent` (полностью под
  контролем неаутентифицированного клиента) в 2048/1024 символа = ~6 КБ/3 КБ.
- **Воспроизведено:**
  ```
  SQLSTATE[54000]: index row size 6160 exceeds btree version 4 maximum 2704
  for index "referrers_value_unique"
  ```
- **Влияние:** `DictionaryValueResolver::resolveId()` бросает исключение → `RecordClickJob`
  падает все 3 попытки → **клик теряется навсегда и молча** (только warning-лог). Тривиальный
  curl-цикл с заголовком сбрасывает все клики целевой ссылки. Codex подтвердил независимо
  (+ пропущенный пункт: `urls.value` — тот же баг через API-ввод).
- **Рекомендация:** резать значения справочников по **байтам**, не символам (`mb_strcut`
  до ~700 символов / 2700 байт), **или** дедуп по `sha256(value)` unique-колонке с полным
  значением в неиндексируемой колонке. Регресс-тест с 2048-символьным CJK-referrer.
- **Файлы:** `app/Services/Links/Clicks/ClickRecorder.php`, `DictionaryValueResolver.php`,
  миграции `..._000003/000007/000008`.

### C2 — Секреты, запекаемые в Docker-образ; реальный пароль БД на диске
- **Категория:** Security / DevOps · **Приоритет:** немедленно
- **Описание:** `Dockerfile:27` — `COPY . /app` **без `.dockerignore`** (подтверждено: файла
  нет). В слои образа запекаются `.env`, `.env.production`, `.env.backup` (с **реальным
  `DB_PASSWORD`** и `APP_KEY`) и вся история `.git` — до `composer install`. Рантайм-`.env`
  монтируется поверх, но запечённые копии извлекаются из любого слоя.
- **Влияние:** доступ к образу/реестру → восстановление прод-кредов БД, `APP_KEY` (он же соль
  Hashids — см. H4), Sentry DSN.
- **Рекомендация:** добавить `.dockerignore` (`.git`, `.env*`, `tests`, `node_modules`,
  `storage/logs`, `docs`); **ротировать `APP_KEY` и `DB_PASSWORD`** (ротацию APP_KEY
  согласовать с H4 — она меняет все коды); запинить базовый образ по digest.
- **Файлы:** `Dockerfile`, корень репо (нет `.dockerignore`), `.env.backup`.

## 🟠 High

### H1 — Баг контракта API: `condition: null` отдаётся как `{"type":null,"data":null}`
- **Категория:** API Contract · **Приоритет:** High
- **Описание:** `RuleResource::toArray()` делает
  `'condition' => ConditionResource::make($this->resource->condition)` без null-гарда. Для
  правила без условия это **не** сворачивается в `null` при вложении — идёт разыменование
  `null->type`/`null->data`.
- **Воспроизведено:** вывод `{"url":...,"condition":{"type":null,"data":null},...}` + два
  PHP-warning. Противоречит `SDK_API.md` (гарантирует `condition: null`) **и** утверждению
  прошлого ревью «проверено: возвращает null» (`docs/CODE_REVIEW.md:12-14`), которое
  **доказуемо неверно**. Codex согласен.
- **Влияние:** каждый SDK-клиент, парсящий `rules[].condition` для безусловных правил,
  получает битую форму.
- **Рекомендация:**
  `'condition' => $this->resource->condition ? ConditionResource::make($this->resource->condition) : null`.
  Добавить тест контракта 201 (T10) на этот кейс.
- **Файлы:** `app/Http/Resources/RuleResource.php:23`.

### H2 — Нет unique-индекса на `callbacks.click_id` → дубли вебхуков при гонке
- **Категория:** Reliability / Data Integrity · **Приоритет:** High
- **Описание:** `CallbackDispatcher` полагается на
  `Callback::firstOrCreate(['click_id' => …])` + диспатч по `wasRecentlyCreated` ради «один
  колбек на клик», но миграция `..._000011` ставит только `index('status')` — **unique на
  `click_id` нет**. `firstOrCreate` без него не race-safe: два конкурентных `RecordClickJob`
  для одного клика могут оба сделать INSERT, оба увидеть `wasRecentlyCreated=true` и **оба
  поставить вебхук**. Codex согласен и отметил: комментарий в `ClickRecorder` «firstOrCreate
  is race-safe on its own» — то самое заблуждение (клики безопасны **только** потому, что
  `clicks.uuid` unique).
- **Влияние:** дубли исходящих вебхуков клиентам сверх документированных «≤5 попыток одного
  колбека».
- **Рекомендация:** добавить `unique` на `callbacks.click_id` (заодно даёт недостающий индекс
  для lookup-а `firstOrCreate`). Поправить комментарий.
- **Файлы:** миграция `..._000011`, `CallbackDispatcher.php:33`, `ClickRecorder.php:30-37`.

### H3 — Stored XSS через неэкранированный футер на **неаутентифицированной** странице логина
- **Категория:** Security · **Приоритет:** High
- **Описание:** `resources/views/filament/layouts/simple.blade.php:73` рендерит
  `{!! setting('content.footer') !!}` без экранирования. `content.footer` редактируется
  админом через `SettingResource`. Simple-layout — это **страница логина**, то есть пейлоад
  исполняется у **неаутентифицированных** посетителей.
- **Влияние:** долгоживущий stored-XSS на `/login`, переживающий деплои: оверлеи кражи
  кредов на форме входа, кража cookie, drive-by редиректы.
- **Рекомендация:** `{{ }}` или прогон через allowlist-санитайзер; в идеале не рендерить
  админский HTML на pre-auth layout. Codex согласен.
- **Файлы:** `simple.blade.php:73`, `SettingForm`.

### H4 — `HASHIDS_SALT` по умолчанию = `APP_KEY`
- **Категория:** Security / Reliability · **Приоритет:** High
- **Описание:** `config/hashids.php` → `'salt' => env('HASHIDS_SALT', config('app.key'))`.
  Следствия: (а) ротация `APP_KEY` молча меняет маппинг id↔code — новые ссылки получают
  другие коды, связь невидима; (б) знающий `APP_KEY` может **декодировать любой код в
  последовательный id ссылки** (энумерация / утечка объёма). Codex согласен.
- **Рекомендация:** задать независимый `HASHIDS_SALT` сейчас (до ротации APP_KEY в C2),
  задокументировать связь.
- **Файлы:** `config/hashids.php:13`.

### H5 — Нет индекса `links.service_id` + не-sargable запросы дашборда (перф под нагрузкой)
- **Категория:** Performance · **Приоритет:** High
- **Описание:** на `links` индексы только по `id` и `code` (проверено на живой БД) — **нет
  индекса на `service_id`/`domain_id`** (Postgres не индексирует FK-колонки автоматически).
  Filament фильтрует ссылки по сервису; проверки FK `restrictOnDelete` сканируют. Плюс:
  фильтр дат в `ClicksTable` через `whereDate()` (не-sargable, обходит
  `clicks_created_at_index`), `COUNT(*)` в бейджах/виджетах на **каждой** загрузке страницы,
  `GROUP BY date(created_at)` в `ClicksChart`.
- **Влияние:** латентность списков/фильтров и накладные расходы на страницу растут с размером
  таблиц; seq scan по крупнейшим таблицам.
- **Рекомендация:** добавить `index(service_id)` (или `(service_id, deleted_at)`) на `links`;
  переписать фильтр `ClicksTable` на полуоткрытые границы `>= $date / < $date+1day`; partial
  index `WHERE status='pending'` на `callbacks`.
- **Файлы:** миграция `links`, `ClicksTable.php:64-73`, бейджи `ClickResource`/`CallbackResource`.

### H6 — Дыры деплоя: нет `migrate` при деплое + почта молча уходит в `log`
- **Категория:** DevOps / Reliability · **Приоритет:** High
- **Описание:** `docker-compose.production.yml` гоняет `storage:link`/`optimize`/
  `octane:frankenphp`, но **никогда `php artisan migrate`** → миграции применяются вручную,
  риск дрейфа. Отдельно: `config/mail.php` по умолчанию `log`, а в `.env.production`
  **нет ни одного `MAIL_`-ключа** → письма сброса пароля и уведомления пишутся в лог и
  **не доставляются**. Codex согласен по обоим.
- **Рекомендация:** добавить gated-шаг `migrate --force`; настроить реальный мейлер либо
  задокументировать восстановление только через консоль (`make:filament-user`).
- **Файлы:** `docker-compose.production.yml`, `config/mail.php`, `.env.production`.

### H7 — Потолок `varchar(8)` для кода — необработанный жёсткий стоп
- **Категория:** Reliability · **Приоритет:** High (латентно)
- **Описание:** `Link.code` = `Hashids::encode(id)` (алфавит 52 буквы, мин. длина 5) в
  `varchar(8)`. **Эмпирически:** первый id, чей код достигает 9 символов —
  **52 523 350 144** (~52,5 млрд). `Link::booted()` пишет код через `updateQuietly()`
  **без гарда/try-catch** → на этом id бросает «value too long for varchar(8)» посреди
  транзакции → необработанный 500 при создании. Codex подтвердил точный потолок.
- **Рекомендация:** расширить `code` до `varchar(16)` (дёшево, снимает потолок на весь
  диапазон bigint) и/или ассертить длину кода ≤ ширины колонки с понятной ошибкой.
- **Файлы:** миграция `links`, `Link.php:91-101`, `HashidCodeGenerator`.

### H8 — `POST /api/links` без throttle → DoS амплификацией хранилища
- **Категория:** Security / Reliability · **Приоритет:** High
- **Описание:** на роуте есть `auth:sanctum` + abilities, но **нет `throttle`**. Каждый вызов
  создаёт до 50 правил + url/condition-строки + `callback_data` (≤50). Утёкший/багнутый
  `links:create`-токен амплифицирует хранилище без квоты. Codex согласен.
- **Рекомендация:** `throttle:api-links` по id токена; квота ссылок на сервис за всё время.
- **Файлы:** `routes/api.php`, `AppServiceProvider`.

  Заинлайнить тривиальные, оставить только `HasManyRules` (несёт `orderBy`).

  `ExampleTest`, TODO в `DomainResource::getRelations()`, лишняя ветка `'hashid'` в

## ✅ Сильные стороны (сохранить)

- **Асинхронно-корректный путь редиректа** — контроллер диспатчит `RecordClickJob` и
  немедленно возвращает редирект; аналитика не блокирует посетителя. Самое важное решение,
  сделано правильно.
- **Идемпотентность встроена** — UUID на резолв, прокинутый через ретраи, +
  `firstOrCreate(['uuid'…])` под unique-индексом делают `$tries=3` безопасным, без дублей.
- **Когерентный, тестируемый сервисный слой** — `LinkCreator`, `ClickRecorder`,
  `CallbackDispatcher`, `CallbackDataRenderer`, `LinkRuleResolver`: одна ответственность,
  constructor-injected, в основном `readonly`.
- **Аккуратный SSRF-гард** — allowlist схем + blocklist хостов + отклонение приватных/
  зарезервированных диапазонов по DNS, перепроверка на момент отправки, `allow_redirects=false`
  с документированным обоснованием rebinding; корректная машина статусов
  4xx-permanent/5xx-retry.
- **Чистая Octane-гигиена** — *проверено*: stateless `readonly`-сервисы, per-call резолв
  request/timezone, без статической аккумуляции, без request/config в синглтонах.
- **Правильная форма расширяемости** — strategy/registry условий (интерфейс + авто-`type()` +
  tagged binding) питает и динамическую валидацию по типу, и обобщённый диспатч.
- **Дисциплина дедупа jsonb** — данные условия валидируются хендлером до кодирования;
  *проверено canonical-safe* по unicode/float/порядку ключей.
- **Образцовая дисциплина регрессий** — `tests/Feature/Regressions/` (файл на баг, docstring к
  пункту ревью), задокументированный red-test-then-fix, чистый **базлайн из 69 тестов**.
- **Идеально единообразная структура Filament** — каждый ресурс `Tables/`+`Schemas/`+`Pages/`;
  тщательный i18n с паритетом en/ru; `callback_data` как round-trip JSON-поле; route binding
  корректно снимает SoftDelete-scope, чтобы trashed-ссылки оставались видимы.
- **Сокращение поверхности на роутере** — constraint `[A-Za-z0-9]{5,8}` 404-ит мусорные пути
  до лимитера; двухуровневый per-IP лимит. (И *проверено*: ни один код Hashids не затеняет
  слаг Filament — `decode("links"/"users"/…)` все невалидны.)

## Что проверено вживую (а не только чтением)

- **C1:** `INSERT` несжимаемого 2048-символьного многобайтового значения в `referrers` →
  `SQLSTATE 54000 index row size 6160 exceeds 2704`. Сжимаемый повтор одного символа проходит
  (TOAST-сжатие) — отсюда нюанс «несжимаемый».
- **H1:** `RuleResource::make($ruleBezUsloviya)->toArray()` →
  `{"condition":{"type":null,"data":null}}` + 2 warning. Прошлое «проверено: null» — неверно.
- **H7:** перебор `Hashids::encode(id)` — первый 9-символьный код на id `52 523 350 144`.
- **trustProxies (опровергнуто как Critical):** при топологии nginx-proxy (append XFF)
  `Request::getClientIp()` для `XFF="1.2.3.4, 203.0.113.50"` и доверенного `REMOTE_ADDR`
  вернул **`203.0.113.50`** (реальный peer), а не подделку. Спуф проходит только при прямом
  доступе к контейнеру из доверенной подсети (Case B) — это hardening, не Critical. Codex
  согласен (с поправкой: `172.16/12` покрывает все Docker-bridge-сети — остаточный риск чуть
  шире, чем «только в обход nginx»).
- **Слаги Filament (опровергнуто):** `decode("links"/"users"/"clicks"/"domains"/"services"/
  "settings"/"login"/"horizon")` — все невалидны; ни один id не кодируется в зарезервированный
  слаг. Затенения сгенерированного кода нет.
- **jsonb-дедуп условий (опровергнуто):** re-select по `"data"::jsonb = ?::jsonb` canonical-safe.
- **Octane (опровергнуто):** утечки состояния между запросами в синглтонах/статике нет.
