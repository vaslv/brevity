отсюда в том же коммите, где закрывается последний пункт (долговечные
решения — в [08-decisions.md](./08-decisions.md), остальное сохранит

[10-competitor-catalog-review.md](./10-competitor-catalog-review.md)

## 3. Этап 1 — детекция ботов + слотовые счётчики

**Статус: реализован 2026-07-12** (все 8 пунктов, каждый прошёл ревью
Codex). Осталось выполнить деплой-шаги ниже; после деплоя раздел
удаляется по правилам из readme.

Делается одной связкой, чтобы счётчики сразу рождались с разрезом
«бот / не бот» и не мигрировать их дважды.

### Ключевые проектные решения

**Флаг бота живёт на справочнике `user_agents`, не на клике.**
User-Agent уже нормализован в справочник (`user_agents.value` уникален).
Детекция по UA детерминирована → флаг вычисляется **один раз на уникальный
UA** при создании строки справочника. Клик получает признак через
`userAgent->is_bot`. Клик без UA считается **не ботом**.

**Детекция — `jaybizzle/crawler-detect`** за собственным интерфейсом
`BotDetector` (одна точка замены библиотеки на этапе 3).

**Счётчики — слотовые, per-link, с разрезом по ботам.** Таблица
`link_click_counters (link_id FK, is_bot boolean, slot smallint 1..100,
count bigint)`, unique `(link_id, is_bot, slot)`. Инкремент — UPSERT
(`ON CONFLICT ... DO UPDATE SET count = count + 1`) **в одной транзакции
со вставкой клика** и только при `wasRecentlyCreated` (ретрай
`RecordClickJob` по существующему `uuid` счётчик не трогает). 100 слотов
размазывают конкуренцию блокировок — нет hot row. Итог = сумма слотов.

**Чего счётчики НЕ решают.** Они дают тотальные значения per-link.
Time-windowed запросы дашборда («сегодня», «за неделю», график по дням)
остаются на индексированных COUNT по `clicks(created_at)` — осознанно.
Счётчики закрывают: `clicks_count` в таблице ссылок (сейчас `withCount`,
деградирует с ростом), сортировку по кликам, разрез «без ботов».

### Шаги этапа 1а — детекция

1. `composer require jaybizzle/crawler-detect`.
2. Миграция: `user_agents.is_bot boolean NOT NULL DEFAULT false`.
3. Сервис `App\Services\Links\Clicks\BotDetector` — обёртка над
   CrawlerDetect за собственным интерфейсом.
4. `ClickRecorder`: резолвинг UA — выделенная ветка: при создании новой
   строки `user_agents` проставлять `is_bot`. Паттерн insertOrIgnore +
   SELECT сохраняется (race-safe). Существующие строки не трогаются.
5. Колбеки (контракт, 03-api.md §10): переменная `{{click.is_bot}}`
   (`"true"`/`"false"`) + `is_bot: bool` всегда в корне payload;
   клиентский ключ `is_bot` перекрывается (задокументировать).
6. Backfill: команда `user-agents:detect-bots` — keyset-обход,
   пере-вычисление флага; `Cache::lock` от параллельного запуска.
   Пригодится и при обновлении библиотеки паттернов.
7. Админка: `IconColumn` «бот» + `TernaryFilter` в Clicks; колонка в
   UserAgents. Переводы `lang/{en,ru}`.
8. Явный `Cache-Control: no-store` на всех ответах резолва (302 и
   интерстишалы) — браузерный кэш не должен съедать повторные переходы:
   каждый клик = колбек (обратное решение к RED-02 источника).

Тесты: unit `BotDetector` (пустой UA / браузер / краулер); клик ставит
флаг новой UA-строке и не трогает существующую; payload колбека содержит
`is_bot`, клиентский перекрывается; backfill-команда; заголовок
`no-store` на ответе резолва.

### Шаги этапа 1б — счётчики

1. Миграция `link_click_counters` (схема выше).
2. `ClickCounterIncrementer` (upsert со случайным слотом). Вызов из
   `ClickRecorder::record`, обёрнутого в `DB::transaction` вместе с
   `firstOrCreate` клика; инкремент только при `wasRecentlyCreated`.
3. Модель `LinkClickCounter` + связи трейтами (`app/Models/Relations/`).
4. Команда `clicks:rebuild-counters`: в транзакции TRUNCATE + агрегатный
   `INSERT ... SELECT` (слот при rebuild всегда 1 — распределение нужно
   только живым конкурентным инкрементам). Лок. Это и первичный backfill,
   и реконсиляция на будущее (пригодится при массовом удалении кликов).
5. Потребители: `LinksTable.clicks_count` → сумма из счётчиков
   («всего / без ботов», сортировка); `StatsOverview` — карточка «всего
   кликов» из счётчиков; «сегодня/за неделю» и `ClicksChart` — без
   изменений.
6. Инвариант в 01-architecture.md: все «числа кликов ссылки» в
   списках/сортировках — из счётчиков; счётчик меняется только в
   транзакции с кликом или rebuild-командой.

Тесты: инкремент при новом клике и НЕ-инкремент при ретрае с тем же uuid;
разрез is_bot; rebuild сходится с COUNT (включая клики без UA); колонка
и сортировка в LinksTable.

### Деплой этапа 1

1. Миграции (колонка + таблица).
2. `user-agents:detect-bots` (backfill флагов).
3. `clicks:rebuild-counters` — строго после backfill'а флагов; TRUNCATE
   внутри rebuild'а гарантирует отсутствие задвоений с кликами,
   записанными между деплоем и rebuild'ом.
4. Обновить 03-api.md §10 (уведомить партнёров), 02-glossary.md,
   01-architecture.md.

## 4. Этап 2 — жизненный цикл ссылки + PATCH API

**Статус: реализован 2026-07-12** (все 10 пунктов, каждый прошёл ревью
Codex). После деплоя раздел удаляется по правилам из readme.

Продуктовые решения приняты 2026-07-12 (см.
[08-decisions.md](./08-decisions.md)): `max_clicks` считает все клики
включая ботов; `valid_since` добавляем; вводим `/api/v1`.

**Модель.** `links` + `valid_since timestamptz NULL`, `valid_until
timestamptz NULL`, `max_clicks integer NULL`.

**Резолвинг.** Ссылка «жива», если одновременно: `valid_since` не в
будущем, `valid_until` не в прошлом, сумма счётчиков (все клики, включая
ботов) < `max_clicks`. Неживая = 404, неотличимо от несуществующей: ни
клика, ни колбека. Из-за асинхронной записи кликов лимит слегка
«мягкий» — осознанно: синхронная резервация отклонена (противоречит
асинхронному ядру), погрешность ограничена лагом очереди (секунды) и
документируется в 03-api.md.

**Версия API.** Все новые эндпоинты и изменения контракта — под
`/api/v1` (те же контроллеры, что legacy, плюс новое). Legacy `/api/*`
замораживается: старый формат ошибок, старый набор возможностей,
deprecated в 03-api.md.

**PATCH `/api/v1/links/{code}`** — новый эндпоинт. Редактируемые:
`valid_since`, `valid_until`, `max_clicks`, `title`, `forward_query`,
`callback_data`, правила (заменой набора целиком). Неизменяемые: code,
domain, service. Sentinel-паттерн: «не передано» ≠ «null»; null обнуляет
ограничитель.

**GET `/api/v1/links/{code}`** — чтение состояния ссылки: та же форма,
что ответ POST, плюс `valid_since`/`valid_until`/`max_clicks` и сводка
кликов из счётчиков (total / без ботов).

**Доступ и скоуп токенов.** Новые abilities `links:read` /
`links:update` (выпуск из админки, как `links:create`). Жёсткий
tenant-скоуп: токен видит и правит **только ссылки своего сервиса**;
чужой код → 404 (не 403 — не раскрывать существование кода).
Полноценные роли-фильтры (KEY-02) остаются отложенными.

**Гигиена трафика (TRK-04).** Конфиг-список игнорируемых источников
(IP / CIDR): совпадение → клик не записывается и колбек не отправляется
(проверка в `RecordClickJob` по оригинальному IP). Плюс query-параметр
отключения трекинга (имя в конфиге) для тестовых переходов; из проброса
`forward_query` он вырезается. Сейчас офисные проверки и мониторинг
порождают не только мусорные клики, но и реальные колбеки партнёрам.

**Query визита → колбеки (GAP-03).** Клик сохраняет сырую query string
визита (`visited_query`, обрезка по байтам как у referrer); в
`callback_data` — плейсхолдеры `{{click.query.<param>}}` (пустая строка
при отсутствии). Партнёр получает свои sub_id/utm обратно в постбеке —
сейчас query можно пробросить только в целевой URL, но не в колбек.
Контракт — 03-api.md §10.

**Ошибки API.** Под `/api/v1` — формат RFC 7807 Problem Details со
стабильными машинными кодами (`validation-error`, `link-not-found`, …);
legacy-маршруты сохраняют старый формат до вывода из эксплуатации.

**Чистки** (artisan, локи, keyset):
- `links:prune-dead` — soft-delete ссылок с истёкшим `valid_until` /
  исчерпанным `max_clicks`; `--dry-run`; без флагов-условий не делает
  ничего (защита);
- `ips:prune` — клики старше N дней (default 90, конфиг) получают
  `ip_address_id = NULL`, осиротевшие `ip_addresses` удаляются.
  In-place-маскирование не годится: unique-индекс на `value` даст
  коллизии масок. Scheduler daily;
- `clicks:prune --link=` (+ админ-действие) — удаление кликов ссылки
  (GDPR-запросы, мусор сканеров, TRK-06); транзакционно с rebuild
  счётчиков этой ссылки.

**Админка.** Порог удаления: `requiresConfirmation` с предупреждением при
кликах выше порога (конфиг, default 15); фильтры «скрыть мёртвые»; поля
valid_until/max_clicks в формах.

Открытых вопросов не осталось — все решения зафиксированы в
[08-decisions.md](./08-decisions.md) (2026-07-12).

## 5. Этап 3 — новые условия + A/B-сплит

**Статус: реализован 2026-07-12** (все 8 пунктов, каждый прошёл ревью
Codex). После деплоя раздел удаляется по правилам из readme.

Каркас готов: `ConditionRegistry` + hendler'ы, `ConditionContext` несёт
весь `Request`.

**Шаг 0 — мульти-условия в правиле (RUL-01).** Сейчас у правила ровно
одно `condition_id`; с ростом числа типов И-комбинации нужны сразу
(«device=ios И utm_source=tg»). Модель: pivot `condition_rule`
(many-to-many; правило матчится, когда матчатся **все** его условия;
правило без условий — безусловное, как сейчас). `rules.condition_id`
мигрирует в pivot. API: `condition` → `conditions: []`; одиночный
`condition` продолжает приниматься (обратная совместимость); лимит
условий на правило — валидацией. Делается **до** новых типов — дешевле,
чем мигрировать после.

**Условия** — по чек-листу «Добавление нового типа условия» из
01-architecture.md, в порядке ценности:
1. `after_date` — зеркально `time_before`;
2. `query_param` — точное key=value;
3. `ip_address` — точный / CIDR / wildcard;
4. `device` — android/ios/mobile/desktop/…; матчинг «UA → несколько
   типов» (iPhone = ios И mobile), enum-валидация;
5. `language` — Accept-Language с порогом q ≥ 0.9.

**Детектор устройств**: `matomo/device-detector`; одновременно
**мигрировать `BotDetector` на него и убрать crawler-detect** (одна
зависимость). После миграции — `user-agents:detect-bots` +
`clicks:rebuild-counters` (паттерны отличаются).

**A/B-сплит.** Таблица `rule_variants (rule_id FK cascade, url_id FK,
weight smallint, label nullable)`. Правило выиграло → при наличии
вариантов URL выбирается по весам; `rules.url_id` — дефолт (без
вариантов поведение как сейчас). Sticky — по решению из §1; сплит —
`hash % sum(weights)` (веса не обязаны давать в сумме 100). Клик пишет
фактический `url_id` **и** `rule_variant_id` (nullable FK) — вариант
различим, даже когда варианты ведут на один URL; в колбеке — плейсхолдер
`{{click.variant}}` (label либо порядковый номер — уточнить при
детализации; пусто без сплита).

**API**: `variants: [{url, weight, label?}]` внутри правила в POST и
PATCH. Валидация: weight ≥ 1, ≥ 2 вариантов.

Тесты: unit на каждый hendler; И-семантика мульти-условий (все / не все
матчатся, пустой набор); детерминизм sticky (один ip+ua → всегда один
вариант) + частотный тест по сетке входов; API с вариантами и
`conditions[]` (включая обратную совместимость одиночного `condition`).

## 6. Этап 4 — геолокация кликов

**Статус: реализован 2026-07-12** (все 6 пунктов, каждый прошёл ревью
Codex). Осталось выполнить деплой-шаги (см. [06-deploy.md](./06-deploy.md));
после деплоя раздел удаляется по правилам из readme.

**Резолвинг** — внутри `RecordClickJob` (запись клика уже асинхронна,
редирект не затрагивается). Локальная база MaxMind GeoLite2-City, пакет
`geoip2/geoip2`, ключ в env (синхронно в `.env` / `.env.example` /
`.env.production`).

**Схема**: справочник `geo_locations (country_code, region, city,
UNIQUE по кортежу)` + `clicks.geo_location_id FK NULL`. Координаты /
timezone не храним (потребителя нет).

**Автообновление «по трафику»** (без cron): после клика — асинхронная
проверка возраста базы (> 30 дней → скачать); `Cache::lock`; история
попыток; backoff после серии ошибок. Первичная загрузка —
`geo:download-db` в деплое.

**Догеолокация**: `geo:locate-clicks` — keyset по кликам без локации.
IP может быть уже вычищен `ips:prune` — такие остаются без локации.

Сбои гео никогда не роняют запись клика: пустая локация + warning.

## 7. Этап 5 — гео-условия и гео-аналитика

- Условие `country` (ISO 3166-1 alpha-2). Требует гео **синхронно в
  резолве**: `GeoLocator` вызывается в `ResolveLink` лениво — только если
  у ссылки есть country-правило; результат прокидывается в
  `RecordClickJob`, чтобы не резолвить дважды.
- Админка: страна/город в Clicks (фильтры), виджет «клики по странам»
  с разрезом «без ботов».
- Колбеки: переменные `{{click.country}}`, `{{click.city}}` (контракт).

## 8. Этап 6 — репутация доменов

**Not-found редиректы.** Три типа промаха: корень домена / невалидный или
мёртвый код / прочий 404. Настройки: `root_redirect_url`,
`invalid_code_redirect_url`, `not_found_redirect_url` на `domains` +
глобальные значения в настройках приложения; приоритет: домен → глобально
→ 404. Плейсхолдеры `{DOMAIN}` / `{ORIGINAL_PATH}`.

**Orphan-визиты** — **отдельная таблица** `orphan_visits (type, domain_id,
path, referrer_id, user_agent_id, ip_address_id, redirect_url,
created_at)`, не `clicks` с nullable FK: у промаха нет ни link, ни
service. `redirect_url` — куда реально перенаправили промах (NULL, если
отдали 404): видно, как сработали not-found правила и куда утекает
трафик. Запись асинхронно, справочники переиспользуются. Ресурс в админке + фильтр по типу; ретеншн
включить в чистки. Отдельный лимит на запись orphan — решить при
детализации.

**robots.txt** — роут с `Disallow: /` (ссылки не для индексации).

**Safe Browsing.** Проверка целевого URL при создании (асинхронный job) +
периодическая перепроверка активных URL (батчи Lookup API). Флаг на
`urls` (`flagged_at`, `flag_reason`). **Реакция на флаг — открытый
вопрос**; рекомендация: не авто-блокировать, а уведомление + бейдж в
админке (ложные срабатывания дороже). Гигиена fetch'ей: таймауты, без
редиректов; `CallbackUrlGuard` как образец.

## 9. Сквозные правила для всех этапов

- Каждый этап: детальный план → подтверждение → реализация, один коммит
  на логический шаг.
- Инварианты: сбой интеграции/обогащения не влияет на редирект и запись
  клика; массовые операции — keyset + лок; связи моделей — трейтами.
- Контракты: изменение payload'а колбека или API — правка 03-api.md в том
  же коммите; термины — 02-glossary.md; подсистемы — 01-architecture.md.
- env-ключи — синхронно в `.env` / `.env.example` / `.env.production`.
- Тесты на каждый шаг, `sail bin pint --dirty` перед финализацией.

импорт из внешних шортенеров; CLI-паритет; Matomo; domain fallback
(конфликтует с моделью ротации); теги (TAG-* — группировка кампаний
живёт в клиентском `callback_data`; вернуться при потребности в
per-campaign аналитике админки); конверт пагинации (API-03 — до
появления list-эндпоинтов); CSV-экспорт кликов (ANA-03 —
Filament-штатно, по запросу); углублённый health-check (API-06 —
`/up` уже есть). Обоснования — в
[10-competitor-catalog-review.md](./10-competitor-catalog-review.md).

Из аудита 2026-06 (реестр находок — в [08-decisions.md](./08-decisions.md)):
ролевая модель за флагом `users.is_admin` — вводить при появлении
недоверенных операторов; retention `clicks`/`callbacks` — частично
закрывается чистками этапа 2, полная политика — отдельным решением.

Кросс-ревью 2026-07-13 (реестр r20–r33 — в
[08-decisions.md](./08-decisions.md)) полностью отработан: все пункты
реализованы, план правок закрыт.

Из кросс-ревью 2026-07-12 (реестр r1–r19 — в
[08-decisions.md](./08-decisions.md)), масштабируемость под большой
`clicks` — вернуться при росте нагрузки:

- **PATCH `rules` у ссылки с историей A/B (r16).** `$link->rules()->delete()`
  каскадом обнуляет `clicks.rule_variant_id` по всей истории в транзакции
  запроса и стирает A/B-атрибуцию задним числом. Варианты: деноормализовать
  метку варианта на клик (сохранить атрибуцию) либо detach вместо delete;
  батчить обнуление вне лока. На текущем масштабе не больно.
- **env-конфигурируемый `trustProxies` (r17).** Список подсетей прокси —
  в env, чтобы прод сужал его без правки `bootstrap/app.php`.
- **`clicks:rebuild-counters` без глобального лока (r15).** Агрегат в
  staging + короткий swap — только если станет узким местом; сейчас
  ОТКЛОНЁНО (текущий `ACCESS EXCLUSIVE` и есть гарантия, что инкременты
  между снапшотом и TRUNCATE не потеряются).
- **Деноормализация `is_bot` на `clicks`.** Если админ-фильтр ботов на
  `clicks × user_agents` станет медленным — снять флаг на клик при записи
  + частичный индекс.
- **Частичные индексы lifecycle на `links`** (`valid_until`, `max_clicks`)
  под `links:prune-dead` и админ-фильтр «живых» — при росте числа ссылок.

## 11. Кросс-ревью 2026-07-14 — план правок (r34–r48)

Пять параллельных ревью-агентов по всему `app/` (HTTP hot-path, сервисы
Links, джобы/гео/консоль, 16 моделей + 37 миграций, Filament/провайдеры).
Каждая находка верифицирована по исходникам и миграциям. Решения по
спорным пунктам приняты владельцем 2026-07-14 (задачи с `*` требовали
выбора варианта — выбор зафиксирован ниже). Реализация покоммитно, один
коммит на находку, без пуша до отдельной команды. После реализации
реестр уходит в [08-decisions.md](./08-decisions.md), раздел удаляется.

### P1 — критично

**r34. Из админки удаляются пользователи → полный лок-аут.**
`EditUser::getHeaderActions()` держит `DeleteAction`, хотя `UsersTable`
документирует инвариант «users are never deletable from the panel».
Регистрация/сброс пароля на панели выключены → удаление последнего
пользователя необратимо без консоли. **Фикс:** убрать `DeleteAction` из
`EditUser` (табличный guard уже это обеспечивает). Тест: на EditUser нет
delete-действия.

**r35. Нормализация URL переполняет `urls.value` → 500 вместо 422.**
Байтовый кап (2000) в `StoreLinkRequest` считается по сырому вводу, а
`LinkRuleSetWriter::resolveUrlId` percent-энкодит URL уже после валидации
и пишет в `varchar(2048)`. Кириллический/CJK путь ~680+ символов проходит
валидатор, после `normalize()` раздувается втрое → `SQLSTATE 22001`,
неперехваченный `QueryException`. **Фикс:** нормализовать до кап-проверки
в `prepareForValidation`/кастомном правиле реквеста (чистый 422) +
защитная перепроверка `strlen($normalized) <= 2000` в writer как
страховка. Тест: длинный кириллический URL даёт 422, не 500.

### P2 — важно

**r36.\* SSRF через DNS-rebinding в колбэках.** `CallbackUrlGuard`
резолвит хост и проверяет IP, но `SendCallbackJob` шлёт на hostname —
Guzzle резолвит повторно; домен с коротким TTL может подсунуть внутренний
адрес. Вектор admin-only (`callback_url` только из админки).
**Решение: дешёвый компромисс** — `force_ip_resolve=v4` +
перепроверка, без полного пиннинга. Тест: колбэк с v4-форсом; guard
по-прежнему режет литеральные приватные IP.

**r37.\* Широкие trusted-proxy CIDR → спуфинг XFF/Host.**
`bootstrap/app.php` доверяет всему RFC1918 → подмена `$request->ip()`
(IP клика, `{{click.ip}}`), обход rate-limit. **Решение: env
`TRUSTED_PROXIES` сейчас** (закрывает отложенный r17) — список подсетей
в env, дефолт узкий/пустой, прод задаёт реальный диапазон LB в
`.env.production`. env-ключ синхронно в `.env`/`.env.example`/
`.env.production`. Тест: доверенный/недоверенный прокси, парсинг env.

**r38. Браузерная таймзона мертва: EncryptCookies обнуляет `tz`.**
Панель под группой `web`, `EncryptCookies` заменяет нерасшифруемый `tz`
на `null` → `FilamentTimezone` всегда `null`, даты в UTC, `ConditionForm`
парсит время условий как UTC. **Фикс:** `$middleware->encryptCookies(
except: ['tz'])` в `bootstrap/app.php`. Тест: cookie `tz` доходит до
`FilamentTimezone`.

**r39. `DeleteAction` на Service без обработки restrict-FK → 500.**
Голый `DeleteAction` в `EditService`, тогда как bulk-путь использует
`RestrictedDeleteBulkAction` (у `links.service_id`/`clicks.service_id`
`restrictOnDelete`). Удаление сервиса с кликами → Postgres 23503 → 500.
**Фикс:** общий `RestrictedDeleteAction` (23503 → уведомление), применить
в хедере `EditService`. Тест: удаление сервиса с ссылками даёт
уведомление, не исключение.

**r40. JSON-настройка сохраняется без валидации, читается с
`JSON_THROW_ON_ERROR`.** Ветка `json` в `SettingForm` без
`->rules(['json'])` (в отличие от `LinkForm`). Опечатка сохраняется
молча → `JsonException` у первого потребителя. **Фикс:** добавить
`->rules(['json'])`. Тест: невалидный JSON даёт форм-ошибку.

**r41. Панель роняет `AuthenticateSession` → смена пароля не убивает
чужие сессии.** Список middleware `MainPanelProvider` (`['web',
EnsureTechnicalHost]`) не содержит `Filament\Http\Middleware\
AuthenticateSession` из стокового стека. **Фикс:** добавить
`AuthenticateSession::class` в `authMiddleware()`. Тест: смена пароля
инвалидирует вторую сессию.

**r42.\* Свежая geo-база не подхватывается Octane-воркерами.**
Click-воркеры не рециклятся (`maxTime=0, maxJobs=0`), `Reader`
кэшируется на всю жизнь воркера → новый `.mmdb` игнорируется до деплоя.
**Решение: умеренный `maxTime`/`maxJobs`** на click-supervisor в
`config/horizon.php`. Тест/проверка: воркер рестартится в пределах окна.

**r43.\* `clicks.created_at` = время джобы, а не визита.** В
`firstOrCreate` нет `created_at`, payload джобы без времени визита →
отставание минуты-часы искажает аналитику и cutoff `ips:prune`.
**Решение: чинить** — захватывать `now()` в `ResolveLink` при dispatch,
пробрасывать в `RecordClickJob`, выставлять `created_at` явно (isset-safe
для старых payload). Тест: клик пишет переданное время визита.

**r44/r45.\* Ключи query-param с точкой/пробелом/скобкой.** PHP mangl'ит
`sub.id → sub_id`: `QueryParamConditionHandler` никогда не матчит такие
ключи, `ResolveLink::forward_query` портит их в целевом URL.
**Решение: чинить оба** — сырой парсинг query-string (как
`CallbackDataRenderer::queryVariables`) в обоих местах. Тест: условие
`sub.id=abc` матчит; forward_query сохраняет `sub.id` в Location.

**r46.\* Поиск в таблице кликов веером `ILIKE` по 6 связям → self-DoS.**
Шесть `searchable()` relation-колонок → OR'ные `whereHas` с
leading-wildcard по неограниченной `clicks`. **Решение: оставить
searchable только `link.code` и `ipAddress.value`**, остальные
(`url.value`/`referrer.value`/`userAgent.value`/`service.name`) увести в
фильтры или `searchable(false)`.

### P3 / уборка

**r47. Деление на ноль в `RuleVariantSelector`.** `crc32($key) %
$totalWeight` — недостижимо через API (`weight min:1` + `CHECK`), но
через админку/сырой SQL с весом 0 → 500 на hot-path. **Фикс:** при
`$totalWeight <= 0` фолбэк на `$rule->url` (fail closed). Тест: вес 0 не
роняет резолв.

**r48. Мёртвая команда-заглушка `app/Console/Commands/V.php`** — пустой
`handle()`, описание «Command description». Не баг; удалить как
scaffold-остаток (по согласованию).

### Порядок реализации

P1 (r34, r35) → SSRF/прокси (r36, r37) → остальные P2 в порядке реестра
→ P3/уборка. Каждый пункт: код + тест, `sail bin pint --dirty`, отдельный
коммит. Пуш — по отдельной команде владельца.
