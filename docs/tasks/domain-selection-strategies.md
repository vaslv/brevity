# Стратегии выбора домена при создании ссылки

## Задача

При создании ссылки через `POST /api/links` клиент может **не** указывать
конкретный домен и при этом получить ссылку на домене **не по умолчанию** —
выбранном автоматически по заданной стратегии. Выбор должен идти либо в
рамках указанной группы доменов, либо, если группа не указана, среди всех
доменов.

Нужно реализовать три стратегии:

- **`random`** — случайный домен из доступных.
- **`round_robin`** — домены выдаются по кругу: каждой следующей ссылке —
  следующий домен.
- **`coldest`** — самый «холодный» домен: тот, что реже всего использовался
  за определённый период (по статистике создания ссылок).

## API

Новые необязательные поля запроса `POST /api/links`:

| Поле | Тип | Примечание |
|---|---|---|
| `domain` | string\|null | Явный домен (как раньше). Взаимоисключим со стратегией. |
| `domain_strategy` | string\|null | `random` / `round_robin` / `coldest`. |
| `domain_group_id` | int\|null | Ограничить выбор группой. Требует `domain_strategy`. |

Правила валидации:

- `domain` **prohibits** `domain_strategy` и `domain_group_id` — явный домен
  и автоподбор нельзя смешивать (иначе `422`).
- `domain_strategy` ∈ `{random, round_robin, coldest}`; **required_with**
  `domain_group_id` (группа без стратегии бессмысленна → `422`).
- `domain_group_id` — `exists:domain_groups,id`.

Порядок разрешения домена в `LinkCreator`:

1. явный `domain` → он;
2. `domain_strategy` → подбор по стратегии (в группе или по всем);
3. иначе → домен по умолчанию (если помечен);
4. иначе → без домена (резолв через `APP_URL`).

Если стратегия задана, но пул пуст (нет доменов / в группе нет доменов) —
`422` (ошибка на `domain_strategy` или `domain_group_id`), а не молчаливый
фолбэк на домен по умолчанию.

Ответ не меняется по форме: фактически выбранный домен виден в
`data.domain` и `data.url`.

## Семантика стратегий

Пул = домены группы (`domain_group_id`) либо все домены. Дальше:

- **random** — `inRandomOrder()->first()`.
- **round_robin** — наименее недавно назначенный домен: сортировка по
  `MAX(links.created_at)` возрастанию, `NULLS FIRST` (ни разу не
  использованные — первыми), тай-брейк по `id`. Каждая новая ссылка делает
  свой домен «только что использованным», поэтому запросы идут по кругу.
  Состояние **не** хранится отдельно — порядок выводится из истории ссылок,
  так что он корректен при добавлении/удалении доменов.
- **coldest** — домен с наименьшим числом ссылок за окно
  `config('domains.coldest_period_days')` (по умолчанию 30 дней). Сортировка
  по счётчику за период возрастанию, тай-брейк по `id`.

Во всех стратегиях soft-deleted ссылки не учитываются (связь `links`
применяет SoftDeletes-скоуп). Статистика **глобальная** — по всем сервисам,
а не по сервису-владельцу токена (домен — общий ресурс; балансируем его
суммарную нагрузку).

## Архитектура

Зеркало паттерна Conditions (tagged-сервисы + реестр + провайдер):

- `App\Services\Links\Domains\DomainSelectionStrategy` — enum значений.
- `App\Services\Links\Domains\Strategies\DomainSelectionStrategyHandler` —
  интерфейс (`select(Builder $pool): ?Domain`, `static strategy()`).
- `RandomDomainStrategy` / `RoundRobinDomainStrategy` / `ColdestDomainStrategy`
  — реализации.
- `DomainStrategyRegistry` — мапа `strategy → handler` (из tagged
  `domain.strategy`).
- `DomainSelector` — строит пул (группа/все) и делегирует хендлеру.
- `DomainStrategyServiceProvider` — тегирует хендлеры, биндит реестр;
  зарегистрирован в `bootstrap/providers.php`.

`LinkCreator` инжектит `DomainSelector` и зовёт его в `resolveDomainId()`.

Точка расширения: добавить стратегию = реализовать handler + добавить тег в
провайдер + значение в enum.

## Конфигурация

- `config/domains.php` → `coldest_period_days` (env `DOMAIN_COLDEST_PERIOD_DAYS`,
  по умолчанию 30).

## Принятые решения по умолчанию (можно пересмотреть на ревью)

- **Статистика глобальная**, не пер-сервис. Если нужна ротация в рамках
  сервиса — добавить фильтр по `service_id` в стратегии.
- **round_robin без курсора** (выводится из истории). Под нагрузкой два
  одновременных запроса могут попасть на один домен — best-effort, ротация
  самокорректируется на следующем запросе. Если нужна строгая гарантия —
  отдельная таблица-курсор + `lockForUpdate`.
- **Период coldest — конфиг**, не настройка в админке и не параметр запроса.
- **`domain_group_id`** ссылается на id (как фильтр `GET /api/domains?group_id=`),
  не на имя.

## Не входит в этот этап

- Стратегия по умолчанию на уровне сервиса/группы (сейчас стратегия задаётся
  в каждом запросе).
- Веса доменов, «прогрев», исключение «сожжённых» доменов.
- Управление стратегией из админки.
